# Сервис заявок на ремонт

Минимальное веб-приложение на Go, PostgreSQL и Docker для обработки заявок на ремонт..

## Что реализовано

- аутентификация по email и паролю;
- роли: заказчик, бухгалтер, рабочий;
- регистрация новых заказчиков;
- создание заявки заказчиком;
- маршрут заявки по статусам:
  1. заказчик создает заявку и отправляет бухгалтеру;
  2. бухгалтер указывает стоимость и отправляет заявку заказчику;
  3. заказчик либо согласовывает заявку и передает рабочим, либо возвращает бухгалтеру;
  4. рабочий берет заявку в работу;
  5. рабочий завершает заявку;
- история движения заявки;
- каталог услуг в отдельной таблице `services`;
- выбор нескольких услуг одного вида через количество в заявке;
- вложения к заявке: `JPG`, `PNG`, `PDF`;
- загрузка файлов как при создании заявки, так и на странице заявки;
- скачивание вложений только после проверки прав доступа;
- мягкое удаление вложений через `deleted_at` и `deleted_by`;
- удаление вложений только на статусе, доступном текущей роли;
- ограничения на загрузки:
  - до `5` файлов за запрос;
  - до `10 MiB` на файл;
  - до `50 MiB` суммарно на запрос;
- Docker Compose для запуска приложения и PostgreSQL.

## Как хранятся данные

- файлы лежат на диске в каталоге `uploads`;
- в PostgreSQL хранятся только метаданные вложений;
- каталог услуг хранится в таблице `services`;
- выбранные в заявке услуги хранятся в `request_services`;
- миграции встраиваются в бинарник через `go:embed` и автоматически выполняются при старте приложения.

Текущий набор миграций рассчитан на развертывание схемы с нуля. Если у вас уже есть старая локальная база данных из предыдущих версий проекта, для dev-сценария проще пересоздать volume:

```bash
docker compose down -v
docker compose up --build
```

## Безопасность

- SQL-запросы выполняются через параметризованные аргументы;
- HTML рендерится через `html/template`;
- пароли хранятся как `bcrypt`-хэши;
- в cookie хранится только сессионный токен, на сервере используется его `SHA-256`-хэш;
- для `POST`-действий используется `CSRF`-токен;
- роли и права проверяются на сервере;
- включены базовые security headers;
- есть простой rate limit на попытки входа;
- для вложений используется whitelist типов и случайные имена файлов.

## Запуск через Docker Compose

```bash
docker compose up --build
```

После запуска:

- приложение будет доступно на `http://localhost:8080`;
- PostgreSQL будет доступен на `localhost:5434`;
- данные базы лежат в volume `pgdata`;
- загруженные файлы лежат в volume `uploads`.

## Локальный запуск без Docker

Нужен PostgreSQL и корректный `DATABASE_URL`. Если база поднята через `docker compose`, то с хоста она доступна на порту `5434`.

Пример для PowerShell:

```powershell
$env:DATABASE_URL="postgres://app:app@localhost:5434/repair_requests?sslmode=disable"
$env:DEV_SEED_USERS="true"
go run ./cmd/server
```

## Тестовые пользователи

При `DEV_SEED_USERS=true` при старте создаются тестовые учетные записи:

| Роль | Email | Пароль |
|---|---|---|
| Заказчик | `customer@example.com` | `password123` |
| Бухгалтер | `accountant@example.com` | `password123` |
| Рабочий | `worker@example.com` | `password123` |

## Как проверить сценарий

1. Войти как `customer@example.com`.
2. Создать новую заявку.
3. Выбрать одну или несколько услуг, при необходимости указать количество больше `1`.
4. Прикрепить файлы `JPG`, `PNG` или `PDF`.
5. Выйти и войти как `accountant@example.com`.
6. Открыть заявку, указать стоимость и отправить заказчику.
7. Войти как заказчик и либо согласовать заявку, либо вернуть бухгалтеру.
8. После согласования войти как `worker@example.com`.
9. Взять заявку в работу и затем завершить ее.

## Основные таблицы

- `users` — пользователи;
- `sessions` — серверные сессии;
- `repair_requests` — заявки;
- `request_events` — история переходов по статусам;
- `services` — справочник доступных услуг;
- `request_services` — выбранные услуги в заявке;
- `request_attachments` — метаданные вложений и поля soft delete.

## Структура проекта

```text
cmd/server/main.go              точка входа
internal/config                 конфигурация из env
internal/database               подключение к PostgreSQL, миграции, seed-пользователи
internal/app                    HTTP-обработчики, роли, статусы, вложения, услуги
internal/database/migrations    SQL-схема
web/templates                   HTML-шаблоны
web/static/style.css            стили
web/static/app.js               JS-отправка POST-форм и уведомления
Dockerfile                      сборка Go-приложения
docker-compose.yml              приложение + PostgreSQL + volumes
```

## Основные переменные окружения

```text
APP_ADDR=:8080
DATABASE_URL=postgres://app:app@db:5432/repair_requests?sslmode=disable
COOKIE_SECURE=false
DEV_SEED_USERS=true
SESSION_TTL_HOURS=168
BCRYPT_COST=12
UPLOAD_DIR=/app/uploads
MAX_UPLOAD_BYTES=10485760
MAX_UPLOAD_TOTAL_BYTES=52428800
MAX_UPLOAD_FILES=5
```
